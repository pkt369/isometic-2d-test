# 글쓰기 스타일

내 글을 대신 쓰거나 다듬을 때 따를 기준이다. 핵심은 **내 목소리를 유지하는 것**이다. 일반적인 AI 문체로 덮어쓰지 않는다.

## 적용 우선순위

1. 실제 글 샘플
2. 이 문서의 Before / After
3. 이 문서의 원칙
4. 일반적인 글쓰기 규칙

실제 글 샘플과 규칙이 충돌하면 실제 글 샘플을 우선한다.

## 문맥별 종결어미

- 공식 문서: 습니다체
- 블로그 / 기술 글: 다체
- 목록 / 항목: 개조식 bullet
- 대화형 메모: 짧고 직접적인 문장

## 핵심 원칙

- 결과만 나열하지 않고 메커니즘을 설명한다.
- 의사결정은 트레이드오프 중심으로 쓴다.
- 표준적인 내용은 압축하고, 고유한 문제는 충분히 풀어 쓴다.
- 기본값이나 제약을 의도적 선택처럼 포장하지 않는다.
- 좋은 선택처럼 보이게 꾸미기보다 왜 그렇게 할 수밖에 없었는지 설명한다.

## 피할 표현

- AI스러운 상투어
  - "~을 통해 ~할 수 있었습니다"
  - "다양한"
  - "효율적으로"
  - "사용자 경험을 향상시켰습니다"
  - "생산성을 높였습니다"
  - 의미 없는 연결어
- 근거 없는 자기 평가어
  - "뛰어난"
  - "탁월한"
  - "혁신적인"
  - "강력한"
- 과도한 열거
- 과장된 톤
- 과도하게 자기비하적인 톤
- 기본값, 외부 제약, 시간 부족으로 한 선택을 능동적 설계처럼 포장하는 문장

## 선호하는 설명 방식

좋은 문장은 보통 다음 구조를 가진다.

1. 무엇이 문제였는가
2. 왜 문제가 되었는가
3. 어떤 선택지를 비교했는가
4. 무엇을 선택했고 무엇을 포기했는가
5. 결과가 어떻게 달라졌는가

모든 글에 이 순서를 억지로 넣지 않는다. 다만 문제 해결 글, 회고, 기술 설명에서는 이 흐름을 우선한다.

## 수정할 때의 기준

- 문장을 예쁘게 만들기보다 뜻이 선명해지게 고친다.
- 애매한 형용사는 수치, 조건, 사례로 바꾼다.
- 긴 문장은 자르되, 인과관계가 사라지면 안 된다.
- 표준적인 배경 설명은 줄이고, 내가 겪은 문제와 판단은 남긴다.
- 문체를 매끈하게 만들려고 개성을 제거하지 않는다.

## Before / After

이 섹션은 실제 글 샘플로 계속 보강한다. AI는 규칙 설명보다 이 예시를 더 강하게 따른다.

### Before

> 다양한 최적화를 통해 성능을 효율적으로 개선하여 뛰어난 사용자 경험을 제공할 수 있었습니다.

### After

> [구체적 병목] → [측정값] → [택한 해법과 기각한 대안] → [측정 기반 결과]

실제 문장 샘플은 계속 추가한다.

## 셀프 체크

글을 다듬은 뒤 다음 질문을 확인한다.

- 이 문장은 내가 실제로 쓸 법한가?
- 근거 없는 칭찬이나 포장이 들어갔는가?
- 결과만 있고 원인이 빠졌는가?
- 선택한 이유와 포기한 것이 드러나는가?
- 어디서나 쓸 수 있는 문장이라면 너무 일반적인 글은 아닌가?
