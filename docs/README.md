# backend-wiki

백엔드 개발 지식을 Markdown으로 모아두는 개인 지식 베이스입니다.

블로그, 공식 문서, 기술 인터뷰 같은 원본 자료를 모읍니다.
AI 에이전트가 이를 읽고 재사용할 수 있는 개념, 원칙, 판단 기준 문서로 정리합니다.

특정 프로젝트의 규칙 저장소는 아닙니다.
여러 프로젝트의 `docs/` 품질을 높이는 공통 지식층으로 씁니다.

방식 출처: [Andrej Karpathy, LLM Wiki gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

## 폴더 구조

```
backend-wiki/
├── 00_context/   개인 정적 기준 (글쓰기 스타일, 사고방식)
├── 10_raw/       원본 자료 보관소 (읽기 전용, 수정 금지)
├── 20_wiki/      AI가 정리한 지식층 (실제 위키 본문)
├── AGENTS.md     모든 AI 에이전트의 운영 규칙 (SSOT)
└── CLAUDE.md     Claude 전용 진입 파일, AGENTS.md로 위임
```

### `00_context/`

잘 바뀌지 않는 개인 기준을 둡니다.

에이전트가 응답, 글쓰기 톤을 맞출 때 참고합니다.

### `10_raw/`

블로그, 공식 문서, 기술 인터뷰 같은 원본 자료를 그대로 모아두는 곳입니다.

수정하지 않는 source of truth이며, AI는 읽기만 합니다.

### `20_wiki/`

위키의 본문입니다.

**무조건 `20_wiki/index.md`부터 읽으세요.**
폴더 트리가 아니라 그 목차가 전체 지도이자 유일한 진입점입니다.

폴더는 3개짜리 저장통입니다.
세부 분류는 `index.md` 제목으로 표현합니다.
링크는 `[[파일명]]` 기준이라 경로와 무관합니다.

버저닝이 있는 라이브러리, 프레임워크 문서는 두지 않고 context7으로 조회합니다. 위키엔 시간이 지나도 유효한 원칙, 기준, 선택 이유만 남깁니다.

안에 들어가는 것:

- `principles/`
  원칙, 기준. 아키텍처, 기술 선택 이유. (아직 비어 있음)
- `conventions/`
  개발 컨벤션. 네이밍, 커밋, MR, 코드 리뷰.
- `operations/`
  운영 규약, 가이드. 개발 외 전반 — 위키, 에이전트 운영, 문서 작성, 부트스트랩, 도구 채택.
- `index.md`
  위키 전체 목차이자 유일한 진입점. 모든 작업, 질문 전 먼저 읽습니다.
- `log.md`
  무엇을, 왜 바꿨는지 최신순으로 쌓는 작업 이력.

## 사용 방법

- 사람은 `10_raw/`에 원본 자료를 추가합니다.
- AI 에이전트는 `10_raw/`를 읽고 `20_wiki/`에 정리 문서를 만들거나 갱신합니다.
- 작업 규칙은 `AGENTS.md`가 단일 기준입니다. 작업 전 반드시 먼저 읽습니다.

## 다른 프로젝트와 연동

이 위키의 실체는 `~/backend-wiki` 한 곳에만 둡니다.

다른 프로젝트는 복제본 대신 루트에 `backend-wiki` 심볼릭링크를 만들고, `./backend-wiki/...` 상대경로로 참조합니다.

- 본진 `~/backend-wiki`는 GitLab에 정상 커밋, push합니다.
- 소비 프로젝트의 심볼릭링크는 머신마다 깨지는 포인터입니다.
  해당 프로젝트 git에 올리지 않습니다 (`.gitignore`에 `backend-wiki` 필수).
